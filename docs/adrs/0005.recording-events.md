# Recording Events

We want to store a complex record of the modifications that occur to the
entities in our system. This will be useful for auditing purposes, to
show history of edits views on entities, to power search, and to power undo
and redo operations.

## Per-entity tracking

We will first define `mutation_entity_event`. This table will capture all
relevant metadata about entity mutations across the system.

Each event record captures:

- **Entity metadata**
  - `entity_type`: Type of the entity being modified
  - `entity_ref_id`: Unique identifier for the entity
  - `entity_version`: Version of the entity after the mutation
- **Mutation metadata**
  - `kind`: The operation performed (`create`, `update`, `remove`)
  - `name`: Name of the mutation/method invoked
  - `trace_id`: Identifier for the top-level action (spanning multiple
    services if relevant)
  - `mutation_id`: Identifier for this mutation within a service (may be
  autogenerated)
  - `timestamp`: When the mutation was handled
  - `session_index`: Sequence index for multiple events on the same
  entity in a given mutation
- **Actor metadata**
  - `source`: App or tool that initiated the mutation
  - `context_str`: Identifier for the user (if any) responsible
- **Mutation data**
  - `data`: Serialized arguments or additional data for the mutation

### Core Use Cases

- **Insert Event**: Efficient, real-time insertion of mutation events.
- **Query Events**:
  - Lookup all events for a given `entity_type` and `entity_ref_id`,
  ordered by descending `timestamp`.
  - Lookup all events by `mutation_id`.
- **Audit/Log Tooling**: Since all mutations reference a `trace_id`
  and `mutation_id`, we can aggregate and visualize system changes
  at different granularities.

## Mutation tracking

We will then store a `mutation_invocation_record`. This table captures
the details about a particular invocation of a mutation.

Each mutation record captures:

- `trace_id`: Identifier for the top-level action (spanning multiple
  services if relevant)
- `mutation_id`: Unique ID for this mutation invocation (autogenerated)
- `timestamp`: When the mutation was handled
- `context_str`: Identifier for the user (if any) responsible
   for triggering the mutation
- `name`: Name of the mutation/method invoked
- `args`: Serialized arguments for the mutation
- `result`: Result of the mutation (if captured)

## Event Relationships

- **Trace**: All changes stemming from a single top-level user
  action (system-wide).
- **Mutation**: Changes within a single service resulting from
  a mutation use case inside a trace.
- **Event**: A single change to an entity within a mutation,
  within a trace.

This model enables efficient auditing, log construction, debugging,
and advanced features like history viewing and undo/redo.
