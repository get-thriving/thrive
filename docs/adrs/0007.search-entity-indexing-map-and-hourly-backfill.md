# ADR 0007: Search entity indexing map and backfill cron

## Status

Accepted (2026-04-21)

## Context

Search indexing is driven mainly by post-transaction hooks and a few
background jobs that upsert or remove rows in the Algolia or SQLite search
backend. There is no durable link between a domain crown entity and the
provider-specific object id used in Algolia, and no systematic reconciliation
if indexing fails or diverges.

## Decision

1. **Table `search_entity_indexing_map`**  
   Columns: `workspace_ref_id`, `entity_type`, `entity_ref_id`,
   `last_indexed_time`, `object_id`, with a unique key on
   `(workspace_ref_id, entity_type, entity_ref_id)`. This table is the only
   persistence exposed by a dedicated `SearchIndexingStorageEngine` in
   `jupiter.core.search.indexing_storage_engine` (SQLite implementation in
   `jupiter.core.search.impl.sqlite.indexing_storage_engine`).

2. **Domain model**  
   A frozen dataclass `SearchEntityIndexingRecord` and abstract repository
   `SearchEntityIndexingMapRepository` back the table.

3. **Indexing operations**  
   - `SearchEntityIndexService` loads the crown (and optional `Note`), calls
     `SearchRepository.upsert`, then upserts the map row with the returned
     `object_id` and current time as `last_indexed_time`.
   - Removal uses either a loaded crown (`remove` + map delete) or
     `remove_by_object_id` on the search repository plus map delete when the
     domain row is gone but the map still has a record.
   - `SearchRepository.upsert` returns `str`: Algolia returns its hashed
     object id; SQLite returns `"{EntityClassName}:{ref_id}"`.

4. **Post-transaction and batch jobs**  
   Logged-in mutation post-work and GC / Gen / Stats / schedule sync jobs call
   `SearchEntityIndexService` so the map stays aligned whenever the search
   index is updated.

5. **Entity repositories**  
   `EntityRepository.find_summary` returns `list[EntityIndexingSummary]`
   (`ref_id`, `last_modified_time`, `archived`), implemented once on
   `SqliteEntityRepository` (root tables omit `parent_ref_id`; non-root tables
   filter by the same parent as `find_all`).

6. **Background cron**  
   `SearchIndexBackfillDoAllUseCase` is registered like other background mutation
   crons: APScheduler runs it once per day on a staggered hour slot. For each
   workspace and each `NamedEntityTag`, it loads
   domain summaries, compares them to the map, re-indexes when missing or stale,
   and removes index entries (using stored `object_id`) when the map has rows
   absent from domain summaries.

7. **App component**  
   `AppComponent.SEARCH_INDEX_BACKFILL` identifies the cron in domain context
   and client naming.

## Consequences

- **Operational**: Algolia deletes for orphaned rows use the stored object id;
  SQLite deletes use workspace + type + id (object id is redundant but kept for
  a uniform API).
- **Schema**: New migration must be applied before the WebAPI or CLI run
  against an existing database.
