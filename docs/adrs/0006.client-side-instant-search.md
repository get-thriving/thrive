# Client-side instant search in the WebUI

## Context

The workspace app bar previously embedded a compact GET form that navigated to
a dedicated Search tool page. That split navigation and typing across two
screens. Search is now only the instant overlay: typing should feel immediate,
filters stay available without leaving the current page, and the chrome stays
minimal until the user asks for search.

## Decision

We added an instant-search experience driven by a `SearchWidget` in
`src/core/jupiter/core/search/components/search-widget.tsx`.

### UX

- The app bar shows a **search icon** only (no always-visible query field).
- Clicking the icon opens a **modal dialog** layered above the workspace.
- **Layout:** On small viewports the dialog leaves about **0.5rem** margin on
  each side; on larger viewports it uses a **wide, tall** footprint (roughly
  `min(56rem, calc(100vw - 4rem))` by `min(80vh, …)`) with larger outer margin so
  it reads as a focused overlay rather than a second full page.
- The top row is a **full-width query field** with **Settings** (toggles the
  filter panel) and **Close** (dismisses the modal).
- **Settings** reveals **advanced filters** in
  `search-tool-filter-fields.tsx` (shared building blocks with the search API
  query shape).
- **Results** use `search-tool-results.tsx` (counts, `EntityCard` +
  `SearchMatchLink`).

### Data flow and auth

The auth token for the WebAPI lives in an **http-only session cookie**, so the
browser cannot construct an `ApiClient` with `TOKEN` the way server loaders do.

Instant search therefore uses a **Remix resource route**
`/app/workspace/search-instant`: the **loader** parses query-string parameters,
maps them to `SearchArgs` in the same module, calls `getLoggedInApiClient`,
invokes `apiClient.search.search`, and returns an `ActionResult` JSON payload. The
widget loads that URL via **`useFetcher`** with a **debounced** effect (300ms)
whenever the query or filter inputs change while the dialog is open.

The client widget keeps the resource URL string and `useFetcher` payload typing
co-located in `search-widget.tsx`.

### Wisdom / pitfalls

- **Debounce everything that triggers a fetch** (query and filters), not only
  the text field, so tag and date changes do not spam the server.
- **Do not list `useFetcher()` in `useEffect` dependency arrays** unless you
  want extra runs; Remix documents a stable fetcher identity, but depending on it
  is brittle—prefer listing the actual input state and calling `fetcher.load`
  inside the effect.
- **Empty query:** the resource loader returns a success envelope with
  `result: undefined` and does not call the search use case, so clearing the
  field clears results without an unnecessary API round-trip.
- **Validation errors:** reuse `validationErrorToUIErrorInfo` so `FieldError` /
  `GlobalError` match other workspace forms.
- **Viewport height on mobile:** the workspace already maintains `--vh`; the
  dialog height uses `calc(100 * var(--vh, 1vh) - …)` so the overlay respects
  mobile browser chrome.

## Status

Accepted; implemented in the WebUI and shared `@jupiter/core` search components.
