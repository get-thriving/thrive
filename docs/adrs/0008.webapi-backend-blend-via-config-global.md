# ADR 0008: Backend blend controlled via `Config.global`

## Status

Accepted (2026-05-03)

## Context

Backend processes wire several replaceable infrastructure pieces: authentication
provider, telemetry sink, CRM integration, domain persistence, and search.
Operators need a single, explicit place to choose that **blend** per deployment,
without scattering ad hoc rules across `env`, hosting flags, and implicit
defaults.

## Decision

1. **Control surface**  
   Environment variables live in `src/Config.global` (loaded with other
   universe-wide settings when building global configuration). Together they
   define the **backend blend** for WebAPI, WebUI, API, MCP, and cron workers
   that share the same deployment.

2. **Global scope**  
   Global blend values apply to **all** processes in a deployment that load
   `Config.global`, not per workspace, user, or request.

3. **Variables and allowed values**

   | Variable | Allowed values | Meaning |
   | -------- | -------------- | ------- |
   | `AUTH_PROVIDER` | `local`, `local-google-apple` | Authentication backend |
   | `EMAIL_VERIFICATION_STRATEGY` | `none`, `verify` | Whether new users must verify email (`none` marks them verified at signup) |
   | `TELEMETRY` | `sentry`, `local` | Telemetry sink (e.g. Sentry vs local/no-op style) |
   | `CRM` | `wix`, `noop` | CRM integration vs no-op |
   | `WEBAPI_STORAGE_ENGINE` | `sqlite`, `postgres` | Primary domain persistence engine (WebAPI) |
   | `WEBAPI_SEARCH` | `sql`, `algolia` | Search query / document storage engine (WebAPI) |
   | `WEBAPI_EMAIL_SENDER` | `noop`, `resend` | Email sender for verification messages (WebAPI) |

   Values are lowercase string tokens. Invalid or missing values should be
   rejected at startup with a clear error once wiring reads these variables.

4. **Credentials and URLs**  
   Provider-specific secrets and connection strings remain in service-level
   config (for example `src/webapi/srv/Config.project` and related env). The
   blend flags only choose **which** implementation to construct; they do not
   replace DSNs, API keys, or database URLs.

## Conventions by universe

- **Thrive** (`UNIVERSE=thrive`, hosted on Render): `AUTH_PROVIDER=local-google-apple`,
  `EMAIL_VERIFICATION_STRATEGY=verify`, `TELEMETRY=sentry`, `CRM=wix`,
  `WEBAPI_STORAGE_ENGINE=postgres`, `WEBAPI_SEARCH=algolia`,
  `WEBAPI_EMAIL_SENDER=resend`.
- **Other universes** (not thrive): local-first globals—`AUTH_PROVIDER=local`,
  `EMAIL_VERIFICATION_STRATEGY=none`, `TELEMETRY=local`, `CRM=noop`, plus
  `WEBAPI_STORAGE_ENGINE=sqlite` and `WEBAPI_SEARCH=sql` unless overridden.
- **Dev** (`UNIVERSE=dev`): no fixed convention; teams may set any valid
  combination while developing or testing.

## Consequences

- Operators set blend entries in `Config.global` (or equivalent injected env) for
  each deployment’s intended behavior.
- Composition code should branch on these settings (or on typed values derived
  from `JupiterGlobalProperties` / service properties) so behavior stays aligned
  with the declared configuration instead of re-deriving the same choices from
  unrelated signals.
